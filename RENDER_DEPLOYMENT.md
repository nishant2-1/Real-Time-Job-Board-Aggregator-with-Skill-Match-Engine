# Render Deployment Guide

## Goal

Deploy JobRadar without relying on Docker on your laptop.

Recommended Render shape:

1. one web service for FastAPI
2. one background worker for Celery worker
3. one background worker for Celery beat
4. one managed PostgreSQL database
5. one managed Redis instance
6. one static site for the frontend

## Services To Create

### 1) PostgreSQL

- Type: Render PostgreSQL
- Database name: `jobrador`
- User: generated by Render
- Use the internal connection values in backend environment variables

### 2) Redis

- Type: Render Redis
- Use the internal Redis host and port in backend environment variables
- Use the same Redis instance for cache, deduplication, broker, and result backend

### 3) Backend web service

- Type: Web Service
- Root directory: `backend`
- Runtime: Python
- Build command:

```bash
pip install -r requirements.txt && python -m spacy download en_core_web_sm
```

- Start command:

```bash
alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

### 4) Celery worker

- Type: Background Worker
- Root directory: `backend`
- Build command:

```bash
pip install -r requirements.txt && python -m spacy download en_core_web_sm
```

- Start command:

```bash
celery -A app.tasks.celery_app.celery_app worker --loglevel=info
```

### 5) Celery beat

- Type: Background Worker
- Root directory: `backend`
- Build command:

```bash
pip install -r requirements.txt && python -m spacy download en_core_web_sm
```

- Start command:

```bash
celery -A app.tasks.celery_app.celery_app beat --loglevel=info
```

### 6) Frontend static site

- Type: Static Site
- Root directory: `frontend`
- Build command:

```bash
npm ci && npm run build
```

- Publish directory:

```bash
dist
```

## Backend Environment Variables

Set these on the FastAPI web service, Celery worker, and Celery beat:

```bash
APP_ENV=production
APP_DEBUG=false
APP_HOST=0.0.0.0
APP_PORT=8000

POSTGRES_HOST=<render-postgres-host>
POSTGRES_PORT=5432
POSTGRES_DB=<render-postgres-db>
POSTGRES_USER=<render-postgres-user>
POSTGRES_PASSWORD=<render-postgres-password>

REDIS_HOST=<render-redis-host>
REDIS_PORT=<render-redis-port>
REDIS_DB=0

CELERY_BROKER_URL=redis://<render-redis-host>:<render-redis-port>/0
CELERY_RESULT_BACKEND=redis://<render-redis-host>:<render-redis-port>/1

JWT_SECRET_KEY=<strong-random-value>
JWT_REFRESH_SECRET_KEY=<strong-random-value>
JWT_ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=15
REFRESH_TOKEN_EXPIRE_DAYS=7

CORS_ORIGINS=https://<your-frontend-domain>
ADMIN_EMAILS=<your-admin-email>

ADZUNA_APP_ID=<optional>
ADZUNA_APP_KEY=<optional>
GREENHOUSE_BOARDS=<optional-comma-separated-list>
LEVER_COMPANIES=<optional-comma-separated-list>

RATE_LIMIT_DEFAULT=60/minute
SCRAPE_INTERVAL_MINUTES=30
SCRAPER_TIMEOUT_SECONDS=20
MAX_RESUME_UPLOAD_BYTES=5242880
MATCH_CACHE_TTL_SECONDS=3600
```

## Frontend Environment Variable

Set this on the Render static site:

```bash
VITE_API_URL=https://<your-backend-domain>
```

## What Gets Stored Where

- PostgreSQL stores users, resumes, jobs, saved jobs, and scraper history.
- Redis stores cache, deduplication keys, and Celery queue state.
- Render static site stores the built frontend assets.

## Recommended Bring-Up Order

1. Create PostgreSQL.
2. Create Redis.
3. Create backend web service with environment variables.
4. Create Celery worker using the same environment variables.
5. Create Celery beat using the same environment variables.
6. Create frontend static site with `VITE_API_URL` pointed at the backend.
7. Open `/docs` on the backend and confirm the API is live.
8. Register an admin user email that matches `ADMIN_EMAILS` and open `/admin` in the frontend.

## Smoke Checks After Deploy

Backend:

- `/health`
- `/docs`
- register a user
- login
- upload a resume

Admin:

- open `/admin`
- confirm users, jobs, and scraper runs render

Background jobs:

- trigger scraper from dashboard or admin tooling
- confirm `scraper_runs` increases
- confirm jobs appear in the jobs board

## Notes

- You do not need Docker Desktop on your machine for this deployment path.
- Render manages process isolation for the web service, worker, beat, Postgres, and Redis.
- If you later want a fully containerized cloud deployment, the current repo can still be adapted for that separately.
